# イントロ演出：破片が終盤に画面中央へ吸い寄せられる問題（調査中）

日付: 2026-07-03
状態: **未解決** — 根本原因の分析は進行中。シミュレーション環境は構築済み。

## 症状

イントロ演出（`src/components/introPrismScene.ts`）でプリズムが砕けた後、
画面の端まで飛んでいった破片パーティクルが、演出の最後の一瞬で
不自然に画面中央へ引き寄せられるように見える。

再生方法: DevTools Console で `sessionStorage.removeItem('pp-intro-seen')` を
実行してからリロード（初回アクセス時のみ再生される仕様のため）。

## 試した修正（失敗）

**修正1**: 重力係数を弱め（`-0.55` → `-0.22`）、外向きの線形ドリフト
（`+ uT * 0.18`）を追加した。

```glsl
// 変更前（現行コード）
vec3 p = position + aVel * damp * 1.2 + vec3(0.0, -0.55, 0.0) * uT * uT;
// 修正1（→ revert 済み）
vec3 p = position + aVel * (damp * 1.2 + uT * 0.18) + vec3(0.0, -0.22, 0.0) * uT * uT;
```

→ ユーザー確認で「まだめっちゃ残ってる」。シミュレーション計測でも
**むしろ悪化**（下表）していたため revert した。

## シミュレーションによる計測

ブラウザ確認ができない環境だったため、シェーダーと同一の数式・同一の
カメラ（fov38°, pos(0,1.05,4.9), lookAt(0,0.05,0)）でパーティクル軌道を
Node で再現する計測スクリプトを作成した: [`intro-prism-sim.mjs`](./intro-prism-sim.mjs)

実行: `node docs/superpowers/debug/intro-prism-sim.mjs`
（このマシンでは `node` が nvm4w の node.ps1 シムに解決されて出力が
消えるため `C:\nvm4w\nodejs\node.exe` を直接呼ぶこと）

計測結果（t は破裂からの経過秒。inward% = 画面上で内向きに動いている

## ここまでの分析

- **現行コードは実際に終盤 5〜7 割の粒子が画面上で内向きに動いている**。
  ユーザーの体感と一致する。
- 原因の候補は「重力の2乗項が終盤に支配的になり、全粒子が一斉に
  垂直落下する」こと。カメラが俯角（約11.5°下向き）なので、落下＝
  カメラ平面から遠ざかる成分を持ち、遠近法で画面中央寄りの消失点へ
  収束して見える。
- 修正1で inward% の立ち上がりは遅らせられたが、終盤（t=1.7）は
  78% と悪化。線形ドリフトにより粒子が速度方向の消失点へ収束し続ける
  こと、クリップ（画面外離脱）が急増して残った粒子が中央クラスタに
  偏ること（central% 29%）が要因とみられる。
- 「ふちっこだった粒子」自体の追跡では r は微増〜横ばいが多く、
  単純な「個々の粒子が中央へ戻る」動きだけでなく、**端の粒子が
  クリップ/フェードで消えて中央の粒子だけが生き残る「密度の収縮」**
  も体感に寄与している可能性がある（加算ブレンドのフェードでは
  暗い端から先に消え、明るい中央が最後まで残る）。

## 次のステップ（案）

1. `intro-prism-sim.mjs` のフレーム出力（BMP、`scratchpad` に生成）を
   目視して、「中央へ吸い寄せ」がどの時間帯・どの機構で見えるか特定する。
2. 有力な修正方向:
   - 重力を 2 乗項でなく減衰付き終端速度にして、終盤の一斉落下を防ぐ
   - 粒子ごとに寿命をランダム化し（aPhase 流用）、全体が同時に
     フェードせず散発的に「星が消える」ようにする
   - 速度の z 成分（奥行き方向）を減らし、画面平面方向へ広がる
     割合を増やす
   - `gl_PointSize` の近接爆発（`/max(0.1,-mv.z)`）に上限を設ける
3. 修正はシミュレーションの inward% / central% で事前検証してから
   実機（ブラウザ）で目視確認する。
